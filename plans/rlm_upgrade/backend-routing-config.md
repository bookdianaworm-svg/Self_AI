# Backend Routing Configuration (RLM Sub-Calls)

**Version**: 0.1  
**Purpose**: Define how the system chooses **which backend client** (model / harness) to use for each RLM sub-call, without hardcoding fixed agents.  
**Companion Docs**: `rlm-routing-upgrade-overview.md`, `environment-routing-config.md`.

---

## 1. Concepts & Definitions

### 1.1 Backend Client

A **backend client** is any LLM/harness the orchestrator can call to handle a sub-call. Examples:

- `rlm_internal` — your current RLM-backed model (default)
- `claude_agent` — Claude Agent SDK / Claude Code
- `openai_gpt` — OpenAI-compatible backend
- `local_llama` — Local model (vLLM, llamafile, etc.)

All backends implement a common interface:

```python
class BackendClient(ABC):
    @abstractmethod
    def chat(self, messages: list[dict], tools: Any | None = None) -> dict:
        ...

    @abstractmethod
    def name(self) -> str:
        ...
```

### 1.2 Task Descriptor (Backend-Relevant Fields)

From the full Task Descriptor (see overview doc), the backend router cares about:

```yaml
intent: "web_research" | "code_generation" | "proof_synthesis" | "refactor" | ...
complexity_score: 0.0-1.0
latency_budget_ms: 500-10000
cost_sensitivity: "low" | "medium" | "high"

# Optional hints
token_estimate: 0-100000
safety_level: "low" | "medium" | "high"      # high = more careful backend

# Observed metrics (rolling)
metrics:
  lean_pass_rate_last_50: 0.0-1.0
  avg_iterations_to_pass_last_50: float
  avg_latency_ms_last_50: float
  avg_cost_last_50: float
```

These fields are generated by the orchestrator and maintained in a small metrics store.

## 2. Configuration Schema

### 2.1 Top-Level YAML Schema

```yaml
# backend-routing.yaml

version: 0.1

backends:
  rlm_internal:
    provider: "internal"
    description: "Default internal RLM-backed model"

  claude_agent:
    provider: "anthropic"
    description: "Claude Agent SDK / Code for complex reasoning & code"

  openai_gpt:
    provider: "openai"
    description: "OpenAI GPT model for general tasks"

  local_llama:
    provider: "local"
    description: "Local LLM via vLLM or llamafile"


defaults:
  # Fallback backend if no rule matches
  fallback_backend: "rlm_internal"

  # Global knobs
  global_cost_priority: "medium"        # low | medium | high
  global_quality_priority: "high"       # low | medium | high


rules:
  # Each rule applies when its `when` clause matches the Task Descriptor.
  - name: "cheap_simple_research"
    when:
      intent: "web_research"
      complexity_score: "<0.4"
      latency_budget_ms: ">=2000"
    choose:
      backend: "rlm_internal"

  - name: "deep_research_claude"
    when:
      intent: "web_research"
      complexity_score: ">=0.4"
    choose:
      backend: "claude_agent"

  - name: "proof_synthesis_strong"
    when:
      intent: "proof_synthesis"
      complexity_score: ">=0.3"
    choose:
      backend: "claude_agent"

  - name: "code_gen_medium"
    when:
      intent: "code_generation"
      cost_sensitivity: "high"
    choose:
      backend: "local_llama"

  - name: "code_gen_high_quality"
    when:
      intent: "code_generation"
      cost_sensitivity: "low"
    choose:
      backend: "claude_agent"

  - name: "refactor_quick"
    when:
      intent: "refactor"
      token_estimate: "<2000"
      latency_budget_ms: "<2000"
    choose:
      backend: "rlm_internal"


adaptive_overrides:
  # Adjust backend choice based on *observed* metrics.

  - name: "avoid_low_pass_backend_for_proofs"
    when:
      intent: "proof_synthesis"
      metrics.lean_pass_rate_last_50: "<0.6"
    override:
      backend: "claude_agent"

  - name: "avoid_slow_backend_under_tight_latency"
    when:
      latency_budget_ms: "<2000"
      metrics.avg_latency_ms_last_50: ">3000"
    override:
      backend: "rlm_internal"
```

## 3. Matching Semantics

### 3.1 Condition Syntax

Supported patterns in `when` clauses:

- Exact match: `intent: "web_research"`
- Numeric comparison string:
  - `">0.4"`, `"<=0.7"`, `"<2000"`, `">=2000"`
- Optional field handling:
  - If a field is missing, the condition fails unless `allow_missing: true` is supported in the future.

### 3.2 Rule Application Order

1. Evaluate `rules` in order; first match wins.
2. If no rule matches, use `defaults.fallback_backend`.
3. Apply `adaptive_overrides`:
   - If an override’s `when` matches, it can either:
     - Replace the chosen backend, or
     - Downgrade/upgrade via a small policy (e.g. “next_cheapest”, “next_fastest”).
   - For simplicity, start with direct replacement as shown.

### 3.3 Determinism

- Rules are processed in file order; keep the file stable in version control.
- Always log:
  - Which rule matched
  - Which backend was chosen
  - Any adaptive overrides applied

## 4. BackendRouter Pseudocode

```python
class BackendRouter:
    def __init__(self, cfg: dict, metrics_store):
        self.cfg = cfg
        self.metrics = metrics_store

    def choose_backend(self, desc: dict) -> dict:
        """Return {"backend": backend_id, "rule": name, "overrides": [...]}"""
        features = self._augment_with_metrics(desc)
        backend = self._match_rules(features)
        backend, overrides = self._apply_adaptive_overrides(backend, features)
        return {
            "backend": backend,
            "rule": features.get("_matched_rule", "fallback"),
            "overrides": overrides,
        }

    def _augment_with_metrics(self, desc: dict) -> dict:
        # Fetch rolling metrics for this intent across backends
        metrics = self.metrics.get_for_intent(desc.get("intent"))
        desc["metrics"] = metrics
        return desc

    def _match_rules(self, features: dict) -> str:
        for rule in self.cfg["rules"]:
            if self._matches(rule["when"], features):
                features["_matched_rule"] = rule["name"]
                return rule["choose"]["backend"]
        return self.cfg["defaults"]["fallback_backend"]

    def _apply_adaptive_overrides(self, backend: str, features: dict) -> tuple[str, list[str]]:
        applied = []
        for override in self.cfg.get("adaptive_overrides", []):
            if self._matches(override["when"], features):
                backend = override["override"]["backend"]
                applied.append(override["name"])
        return backend, applied

    def _matches(self, cond: dict, features: dict) -> bool:
        for key, pattern in cond.items():
            value = self._get_nested(features, key)
            if not self._match_single(value, pattern):
                return False
        return True

    def _get_nested(self, obj: dict, key: str):
        # Supports keys like "metrics.lean_pass_rate_last_50"
        parts = key.split(".")
        cur = obj
        for p in parts:
            cur = cur.get(p)
            if cur is None:
                return None
        return cur

    def _match_single(self, value, pattern: str) -> bool:
        if value is None:
            return False
        # Numeric pattern
        if isinstance(value, (int, float)) and isinstance(pattern, str):
            import re
            m = re.match(r"([<>]=?)([0-9.]+)", pattern)
            if not m:
                return False
            op, num = m.group(1), float(m.group(2))
            if op == ">":
                return value > num
            if op == ">=":
                return value >= num
            if op == "<":
                return value < num
            if op == "<=":
                return value <= num
        # Exact match
        return value == pattern
```

## 5. Metrics Store Requirements

The router depends on a **Metrics Store** updated after each sub-call:

For each `(backend_id, intent)` pair, track rolling stats such as:

```yaml
backend_id: "claude_agent"
intent: "proof_synthesis"

lean_pass_rate_last_50: 0.82
avg_iterations_to_pass_last_50: 1.6
avg_latency_ms_last_50: 2800
avg_cost_last_50: 0.045
samples: 50
```

Updates occur after Lean/Haskell verification completes.

## 6. How This Integrates with the Orchestrator

### 6.1 At Sub-Call Creation Time

```python
def run_subcall(prompt: str, desc: dict):
    # desc contains intent, complexity_score, etc.
    route = backend_router.choose_backend(desc)
    backend_client = backend_factory.get(route["backend"], config)

    # Build RLM instance with this backend and whatever environment
    env_id = environment_router.choose_env(desc)
    env = environment_factory.get(env_id, config)

    sub_rlm = RLM(backend=backend_client, environment=env)
    result = sub_rlm.completion(prompt=prompt)

    # After Lean verification:
    metrics_store.update(route["backend"], desc["intent"], verification_result)
    return result
```

### 6.2 No Hardcoded Agents

- The orchestrator never references named agents.
- It only looks at **intent** and other descriptor fields.
- New roles can emerge just by emitting new `intent` values (e.g. `"thermal_optimization"`) and updating the routing config.

## 7. Operational Tuning Guidelines

- Start with simple, human-written rules for 3–4 intents:
  - `web_research`, `code_generation`, `proof_synthesis`, `summarization`.
- Log routing decisions and Lean outcomes to a dashboard.
- After you have data:
  - Increase use of strong backends where they materially improve `lean_pass_rate`.
  - Downgrade to cheaper backends where they perform similarly.

Keep the YAML file **small and readable** at first; let the AI IDE help refactor as patterns emerge.
